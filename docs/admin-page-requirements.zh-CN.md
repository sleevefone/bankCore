## 1. 背景与目标

当前后管页面定位是“测试页”，但核心诉求已经很明确：  
必须把“列表 + 详情”这条最小链路做顺，避免多余操作和错误交互。

本期只做可用性，不追求复杂运营能力。

## 2. 范围（本期）

- 客户主表页, 查询出客户表数据的列表信息，并且展示列表，列表有一列是用操作按钮用来查看具体item的详情的，跳转到详情页面
- 账号主表页, 查询出客户表数据的列表信息，并且展示列表，列表有一列是用操作按钮用来查看具体item的详情的，跳转到详情页面
- 开户操作页（保留）
- 有一个菜单来选择不同的操作， 点菜单应该默认展示首页数据

不在本期范围：

- 权限体系
- 审批流
- 导出/批处理
- 多角色页面拆分

## 3. 用户核心诉求（已确认）

1. [已完成] 切到“客户主表”标签后，能够查询并展示客户列表。  
2. [已完成] 切到“账号主表”标签后，能够查询并展示账号列表。  
3. [已完成] 详情查看应基于“操作列按钮”，不应要求额外输入主键。  
4. [已完成] 页面应保持简单直接，避免“输入框 + 按钮 + 列表 + 详情”重复操作。  
5. [已完成] 列表与详情关系必须清晰：点“查看详情”后跳转到对应详情页。  

## 4. 交互要求

### 4.1 客户主表

- 进入页签后可直接执行列表查询（默认 pageNo=1,pageSize=20）。
- 查询条件：`customerNo/customerName/mobile/pageNo/pageSize`。
- 列表字段：`customerNo/customerName/mobile/idNo/status`。
- 增加操作列“查看详情”按钮：
  - 点击后跳转客户详情页面；
  - 详情页根据 `customerNo` 调用 `/admin/customers/{customerNo}` 展示详情。

### 4.2 账号主表

- 进入页签后可直接执行列表查询（默认 pageNo=1,pageSize=20）。
- 查询条件：`customerNo/accountNo/accountType/pageNo/pageSize`。
- 列表字段：`accountNo/customerNo/accountType/currency/availableBalance/frozenBalance/status`。
- 增加操作列“查看详情”按钮：
  - 点击后跳转账号详情页面；
  - 详情页根据 `accountNo` 调用 `/admin/accounts/{accountNo}` 展示详情。

### 4.3 通用规则

- 禁止“详情手输查询”作为主路径。
- 列表为空时显示明确空态。
- 查询失败时显示错误，不吞异常。
- 菜单用于选择“开户操作页/客户主表页/账号主表页”。

### 4.4 开户页科目输入规则（新增）

- 当前状态：`subjectCode` 已改为下拉选择，并默认带出推荐科目。
- 目标状态：
  - 根据账户类型自动带出默认科目；
  - 前端以“科目下拉选择（编码+名称）”为主，不允许自由输入；
  - 仅在特定权限下允许手工覆盖科目（后续迭代）。
- 验收口径：
  - 默认流程下无需手填科目即可开户；
  - 页面可见科目名称与编码；
  - 下拉中明确标注科目作用。

## 5. 接口约束（已存在）

- 客户列表：`GET /admin/customers`
- 客户详情：`GET /admin/customers/{customerNo}`
- 账号列表：`GET /admin/accounts`
- 账号详情：`GET /admin/accounts/{accountNo}`

## 6. 验收标准（DoD）

1. [通过] 点击“客户主表”标签，能查询并展示列表或空态。  
2. [通过] 点击“账号主表”标签，能查询并展示列表或空态。  
3. [通过] 任一列表点击“查看详情”后，跳转到正确详情页且主键一致。  
4. [通过] 页面中不存在“用于详情查询的主键输入框”。  
5. [通过] 空列表和错误场景下，提示明确。
